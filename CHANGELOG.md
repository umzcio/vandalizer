# Changelog

All notable changes to this project will be documented in this file.

The format is based on [Keep a Changelog](https://keepachangelog.com/en/1.1.0/).

## [Unreleased]

### Added
- **WCAG 2.1 AA accessibility across the app, plus an automated axe-core regression gate.** The frontend now conforms to WCAG 2.1 Level AA, verified over five review-and-fix passes. Highlights: every form control is programmatically labelled; modals trap focus and restore it on close (via `focus-trap-react`) and close on Escape; a global `:focus-visible` keyboard ring is applied consistently (and controls no longer suppress it with inline `outline:none`); a "Skip to main content" link and `<main>` landmark are present; each route sets a descriptive document title; text and icon contrast meets AA on light surfaces (including an accessible-on-light derivation of the brand highlight color); tab bars, radio-card groups, switches, and comboboxes expose correct ARIA roles/state with `aria-controls`↔`tabpanel` wiring and keyboard operation; status changes (toasts, chat, long-running panels) announce via polite live regions; native `confirm()`/`alert()` were replaced with the accessible in-app confirm/toast; personal-data fields carry `autocomplete`; and animations honor `prefers-reduced-motion`. A new Vitest + `vitest-axe` suite (`src/a11y.test.tsx`) asserts zero axe-core violations on representative remediated components so regressions are caught in CI. Findings for each pass are documented in `wcag_report_findings_pass1..5.md`.

## [v4.8.0] - 2026-07-01

### Added
- **Opt-in anonymous deployment telemetry, so the maintainers can see how many self-hosted installs exist and roughly how heavily they're used.** Off by default and privacy-first: when enabled, each deployment sends a once-a-day heartbeat carrying only an anonymous random instance UUID, the version, a coarse environment label, and *bucketed* usage counts (e.g. "11-50 users") — never document content, names, emails, or keys, and every payload is logged locally before it's sent. An optional voluntary identity tier (`TELEMETRY_ORGANIZATION` / `TELEMETRY_CONTACT_EMAIL`) is self-declared only and omitted entirely when blank. Enablement is DB-backed (`SystemConfig.telemetry_config`) so a global admin can opt in or out at runtime — via a new **Telemetry opt-in banner** on the Admin page (shown only while telemetry is off, undecided, and the installer never asked) — without editing `.env` or restarting; the env var is only the initial default. `setup.sh` also asks at install time and now on `--redeploy` / `--upgrade`, reusing the institution name already collected. Separately, a single collector instance can set `TELEMETRY_COLLECTOR_ENABLED=true` to expose the public heartbeat-ingest route and an admin **Telemetry** dashboard (fleet size, version/environment distributions, named deployments); both stay completely hidden on every normal deployment. Full disclosure in [docs/telemetry.md](docs/telemetry.md).
- **"Add Knowledge Base" in the chat composer.** The chat "+ Add" menu gains a third item (alongside Add Document and Add Website) that opens the knowledge-base picker; choosing a KB there activates it for the conversation through the existing flow. No backend change — the chat endpoint already accepted a knowledge base.
- **Staff (analytics/operations role) can reach the tools they need without full admin.** The audit-log query + CSV export, the demo/trial management endpoints, and the System Configuration nav item now allow `is_staff` in addition to `is_admin`, matching how the frontend already gated the Demo tab — so an analytics-role user can do their job without being granted full admin.
- **Prompts and Formatters get their own Library filter chips.** The Library's Mine/Team view previously lumped extractions, prompts, and formatters under one "Tasks" chip, leaving prompts and formatters reachable only via "All". Each now has a dedicated, mutually-exclusive chip with its own count. Frontend-only; no data-model change.

### Changed
- **Warmer, unlimited end-of-trial renewal.** Expired-trial users were dead-ending on a terminal "thank you" page (or a bare landing page) instead of the renewal screen that already existed, and self-extensions were capped at two. The extension cap is removed (renewals are now unlimited; `trial_extensions_used` still increments for analytics only), the GitHub "get in touch" dead-end is gone, and every entry point (login, a mid-session 403, the expiry email) now routes to `/demo/trial-end` with warm copy acknowledging the beta is built with and for trial users.
- **Library terminology standardized on "Extraction(s)".** The Library filter chip read "Tasks" while filtering `set_type='extraction'`, and the create modal/button mixed "Task" and "Extraction". All user-facing Library strings now say "Extraction(s)" and the create button is type-correct (Create Extraction / Prompt / Formatter). Backend `set_type` unchanged.

### Fixed
- **Trial credential recovery was broken in several ways at once.** A cluster of support tickets traced to: the "Resend them" link in trial emails pointing at a route that didn't exist (client-side "Not Found"); activation, per-user resend, and admin bulk-resend each minting a *new* password on every send, silently invalidating earlier emails; activation emails that failed to send being ignored, leaving accounts whose owners never got in; and password reset never clearing a `locked` trial status (which login rejects before checking the password, so reset was futile). Recovery is now magic-link-first — no plaintext passwords emailed, no password rotation on resend, magic-link TTL raised 48h → 14 days — and `resend_credentials` is status-aware (active → fresh link, pending → waitlist message, expired/locked → renewal token routing to `/demo/trial-end`). Email-send failures are now checked and logged.
- **Expired-trial logins dead-ended on a 401 instead of the trial-end screen.** The SPA throws on any non-2xx and never reached its `demo_expired` redirect, so the deployed end-of-trial screen never rendered. `/login` now returns a 200 `demo_expired` payload (no session token) for a locked trial, and the service detects the locked trial before password verification, so the app redirects to `/demo/trial-end`.
- **KB chat silently dropped its grounding guardrails on every follow-up turn.** The grounding prompt was passed as a `system_prompt`, which pydantic-ai only injects on the first request of a run; multi-turn chat rebuilds history from stored message text (no system prompt), so follow-up questions ran with no cite-by-filename / refuse-when-unsupported rules. It's now passed as `instructions`, which pydantic-ai re-applies on every model request.
- **PDF document and URL knowledge-base sources failed to ingest.** A document source read the document's `raw_text` exactly once at "Add Documents" time and errored permanently if extraction (async, OCR-first for PDFs) hadn't finished — so the source never re-ingested once the text filled in. Document sources now park as "pending" while extraction runs and are settled (re-ingested or errored) by the extraction-completion task, preserving page citations. Separately, `fetch_url` decoded every response as HTML, so a direct `.pdf` link yielded nothing; it now sends browser-like headers and, when the response is a PDF (by content-type or `.pdf` extension), extracts via the PDF reader.
- **Out-of-scope KB questions could still get a confident, fabricated answer.** Grounding was prompt-only, so the model could answer from the weakly-related chunks vector search always returns. A deterministic per-KB retrieval-similarity floor (`RAGConfig.min_similarity`) now drops chunks below the threshold before generation; when that empties the candidate set, the existing empty-retrieval path produces a clean "not in the knowledge base" refusal. Ships disabled everywhere (floor `0.0`); turning it on is a deliberate, data-driven step, and live KB chat honors the same per-KB floor as the validation harness.
- **KB "Synthesized Answer" (answer mode) returned a 401 in workflows.** The Knowledge Base Query node was the only LLM-using node not wired to system config in the engine builder, so the synthesis call resolved a `no-api-key` placeholder and routed to the provider's public host. It now receives system config like every other node. ("Passages" mode was unaffected — it makes no LLM call.)
- **KB judge and extraction judge returned a zero-token "Connection error" across Celery event loops.** A cached pydantic-ai Agent carries an httpx pool bound to the loop that built it; reusing a judge agent built during tuning (loop A) on a later "Run Now" validation (loop B) raised "bound to a different event loop" and surfaced as 0% scores / "judge error: Connection error". Both judge caches are now loop-aware (rebuilt when the running loop differs, reused within one run) and are flushed by `clear_agent_caches()` so an admin config save clears them too.
- **A read against a not-yet-ingested knowledge base could create an empty Chroma collection — or fail outright on a read-only store.** Retrieval used `get_or_create_collection`, a write. `query_kb` now uses a read-only lookup that returns `None` when the collection is absent and degrades to no results instead of writing or raising "attempt to write a readonly database".
- **Share-link recipients could open a shared workflow but got a 404 when they ran it.** The run and status-polling paths authorized purely by ownership/team membership and ignored the share token. The token is now threaded through run, batch run, status, batch-status, stream, and cancel (each re-authorizes independently, so patching only `/run` would 404 on the first status poll), with a view-only "Save a copy to my library" affordance. Recipients still run against only their own documents.
- **Examiners got an opaque 404 managing verified knowledge bases.** The frontend enabled manage controls for `is_examiner`, but the backend `can_manage_knowledge_base` honored only owner / team-manager / admin, and the route collapsed "not found" and "not authorized" into one 404. Examiners can now manage *verified* KBs (scoped to verified only), and a new `_require_manageable_kb` distinguishes a genuine 404 from a 403 (logged with context). A "Clone to validate" CTA now gives non-owners an editable copy instead of a disabled button with no path forward.
- **Verified-collection badge counts over-reported.** A collection's `item_ids` is a static curated list that can still point to items since un-verified, deleted, or scoped to an org the viewer isn't in — all dropped when the category opens, so the raw count disagreed with what rendered. Badges now use a `visible_count` computed with the verified + org-visibility filter, and collections with nothing visible to the caller are omitted.
- **The Certification "Batch Processing" module was unpassable for everyone.** Its grader queried `WorkflowResult.user_id`, a field that doesn't exist on the model, raising and surfacing as a generic "cannot be verified, try again later". It now scopes batch results through the user's own `Workflow` ids (the same way every other validator scopes its data). No schema change.

### Operator Notes
- **Five new `TELEMETRY_*` variables in `backend/.env.example`, all safe defaults.** `TELEMETRY_ENABLED` (off), `TELEMETRY_ENDPOINT` (defaults to the maintainers' collector — blank it to hard-disable sending, e.g. for air-gapped installs), `TELEMETRY_PROMPTED` (set true once `setup.sh` has asked, so the in-app banner doesn't re-ask), the optional `TELEMETRY_ORGANIZATION` / `TELEMETRY_CONTACT_EMAIL` identity fields, and `TELEMETRY_COLLECTOR_ENABLED` (turns a deployment into the fleet collector — leave `false` on every normal install). Existing `.env` files predating telemetry inherit the endpoint default so the in-app banner works; nothing is sent until an admin or the installer opts in. See [docs/telemetry.md](docs/telemetry.md).
- **`setup.sh --redeploy` / `--upgrade` now prompt for the telemetry choice** (tty-guarded so it never blocks auto-update/cron, and idempotent — asked at most once, guarded by `TELEMETRY_PROMPTED`), closing the gap where the original install-time prompt only reached new installs. The opt-in is applied on the *same* deploy: `api` + `celery` are recreated once the choice is written, since Compose only re-reads `env_file` on (re)create, not restart.
- **Management `GET /api/mgmt/v1/stats` no longer 500s.** Two independent failures fixed: a `distinct` active-user query was called on a Beanie `FindMany` (no such method) and now routes through the motor collection; and a token-count `$sum` could promote to BSON `Decimal128` (from a single legacy stub document), which `int()` rejected — now coerced defensively. Web-tier Sentry capture was also made deterministic by registering the Starlette/FastAPI integrations explicitly rather than relying on auto-enable ordering.
- **DEPLOY.md gains an LLM provider compatibility table** (OpenAI, Azure OpenAI, Anthropic, OpenRouter, Gemini, Ollama, vLLM, and the generic OpenAI-compatible case), replacing the flat bullet list and fixing the stale protocol list. It explicitly documents that **Microsoft 365 Copilot is not supported** (it's an end-user product, not a model API — use Azure OpenAI for Microsoft-hosted models).
- **No verified-catalog change this release.** `backend/seeds/VERSION` remains **1.1.1**; no re-seed is required when upgrading from v4.7.0.

## [v4.7.0] - 2026-06-22

### Added
- **Projects — a scoped container for one piece of work.** A Project gathers everything for a single grant, contract, or compliance effort in one place: a dedicated folder tree for its files, an implicit knowledge base that auto-ingests every file in the project, the workflows/extractions/automations/knowledge bases you pin to it, and a chat that answers only from the project's documents. Projects are reached from a left-rail **Projects** drawer; the empty state shows an onboarding tutorial (what a project gathers, the four-step setup). Opening a project scopes the whole workspace to it — Files root at the project folder, Chat searches the project KB, and the Automations/Knowledge tabs filter to that project's pins (with a "Show all" escape). A yellow context bar names the active project and offers Manage / Exit. Projects are **personal by default** and never auto-shared; the owner can share with the current team or make it personal again, and can mint revocable **invite links** (viewer or editor, with optional expiry / max uses) for collaborators outside the team. Resources created inside a project (a new KB, a new automation) auto-pin to it, and any catalog item can be pinned from Explore. The active project survives a page reload (the scope is persisted and re-fetched, self-healing if the project was deleted or access revoked), and switching teams clears it. Backed by new `Project` / `ProjectMembership` / `ProjectJoinLink` / `ProjectPin` models and a `/api/projects` router.
- **One unified "Validate & improve" flow across knowledge bases, extractions, and workflows.** The three previously-divergent quality surfaces (KB tuning, extraction validation, workflow validation) now lead with the same **Validate & improve** action and report results on the same scale, with the optimized configuration labeled **Optimized** against your current settings and a no-config baseline. Extraction validation in particular was unified onto the optimizer's closed loop — baseline → judge-scored sweep → apply → post-apply re-validation — and the legacy "Find Best Settings" tuner and divergent score card were removed. A small-test-set discount is applied consistently to both the optimized headline and the post-apply certified score, so applying a better config never *looks* like a regression; the panels disclose the discount and how many test cases clear it.
- **Certified scores measured after apply, plus plain-English trial explainers.** Applying an optimized extraction config now runs a post-apply validation pass on the full test set and writes an authoritative **certified score** to the item's quality tile — no separate validation run needed. Every trial in the results table (KB, extraction, and workflow) is now clickable, opening an explainer modal that spells out, in plain language, the config that trial used, why it was or wasn't selected, and the per-step / per-field outcome breakdown.
- **Cost governance for "Validate & improve" runs.** A per-user concurrency cap (default 3 active runs, counted across KB / extraction / workflow optimizations together; configurable via `SystemConfig.quality_config.max_concurrent_optimizations_per_user`) plus a per-endpoint rate limit (5/min) and an `optimization.start` audit event now guard the optimizer. Exceeding the cap or rate returns a 429 the UI surfaces; super-admins bypass the cap but are still audited. The **Validate & improve** controls are gated on a server-authoritative `can_manage` flag threaded into the extraction, workflow, and KB panels (disabled with a tooltip for view-only users), matching the backend's `manage=True` enforcement.
- **Stale validation-plan detection with one-click regenerate.** A workflow's validation plan can drift out of sync after you edit the workflow's steps. The backend now stamps a canonical definition hash on the plan and flags it stale when the workflow changes (or when a check points at a step that no longer exists). The Validate tab shows an amber banner explaining the drift with two manage-gated actions: **Regenerate plan** (re-derives the auto checks while preserving and re-mapping your manual ones) or **Plan is still correct** (re-stamps the hash without regenerating). Validation responses carry `plan_stale` / `stale_reasons` so the signal is available wherever a plan is read.
- **Choose how a KB evaluation set is built, and edit the questions in it.** The KB Validate & improve wizard's Test-set step now offers three modes — use your saved questions only, generate a fresh set, or combine both — with a coverage level for the generated questions. Generated and saved questions are editable inline in the wizard (and in the standalone Test Queries tab), and you can deselect or permanently remove individual questions before a run; the surviving set is what's actually validated. Saved KB test queries gained a `PATCH …/test-queries/{uuid}` endpoint so per-field edits don't clobber server-owned judge metadata. The same select-and-remove-before-running affordance was added to extraction test cases.
- **Folder actions: Move, Export, Ask, Run, and Add to KB.** The folder context menu in the file browser gained five actions: **Move to…** (reparent a folder, with cycle/ownership guards and the shared team root kept immovable), **Export contents** (download the whole subtree as a structure-preserving `.zip`, reusing per-document authorization), **Ask about folder** (scope chat to just that folder's documents and focus the composer), **Run workflow on folder** (batch-run a chosen workflow over every document in the folder), and **Add to knowledge base** (add the folder's documents to a chosen or newly-created KB). Workflow runs and KB adds now accept `folder_uuids` and expand them server-side to the viewable documents within.
- **Add an entire folder of documents to a knowledge base.** The KB "Add documents" picker now has an **Add entire folder** action with an **Include subfolders** toggle, collecting every non-deleted document in the folder (or subtree) through the normal ingestion pipeline instead of selecting files one at a time.
- **Knowledge Base source provenance and full ingested-text view.** Every KB source now carries an editable **Source** provenance field (defaulting to the origin URL for web sources, free-text for documents), shown in the sources list (URLs linkified) and editable in the source inspector. The inspector also now shows the **complete extracted text** that was actually chunked into the KB — including document sources, which previously showed nothing — so you can see exactly what the KB is answering from.
- **Admin org-wide Knowledge Bases inventory.** A new **Knowledge Bases** tab in the admin panel lists every KB across all users and teams — title, tags, owner, team, source/chunk counts, status, verified flag, and last-updated — searchable and sortable, with inline rename for full admins. Backed by a new `GET /api/admin/knowledge-bases` endpoint that batch-resolves owner emails and team names.
- **Reuse saved Prompts and Formatters in workflow steps.** Prompt and Formatter steps can now link a saved Library prompt/formatter (mirroring "Use Saved Extraction Set"), so a single edit to the saved item propagates to every workflow that references it at runtime. The picker also stops leaking prompts/formatters into the extraction-set chooser.
- **Test Step for API nodes.** The API node joins the task types you can run in isolation from the step editor, so you can debug a request — URL, headers, body, templating — without executing the whole workflow.
- **See each step's last-run output in the workflow editor.** A read-only "Last run output" panel in the step editor shows what a step produced on its most recent run (formatted output, task count, timestamp), so authors can inspect intermediate results without deleting downstream steps or re-running.
- **Editable workflow descriptions, surfaced in the library.** Workflow descriptions are now click-to-edit in the editor (mirroring the title) and shown as a one-line subtitle in the library list, so workflows are self-documenting.
- **Deployment version footer.** A version + environment footer in the sidebar makes deployments visually distinguishable; an optional `DEPLOYMENT_LABEL` setting (falling back to `ENVIRONMENT`) keeps two production deployments from both reading "production", surfaced via a new public `GET /api/config/version`.
- **Feature explainer pills on the Automations and Knowledge browse screens**, so "What are automations?" / "What are knowledge bases?" stays reachable once you already have items, not only from the empty state.
- **Search the admin Demo applications list** by name, title, email, or organization.
- **Creator credit on verified catalog items.** Approving a verification request now stamps a static credit — "by Jane Doe at University of Idaho" — onto the item's catalog metadata, built from the submission's name/org with the deployment's institution (`SystemConfig.org_name`) as the org fallback. The credit is plain text rather than a user reference, so it survives catalog export and seeding to installs where the creator has no account; the Explore/catalog cards fall back from the live submitter to the static credit. The catalog export embeds it, the seeder applies it from `_seed_meta.credit`, and `setup.sh` now asks for the institution name during install (stored as the branding org name when not already set). Ships with catalog **v1.1.1**, which credits the ten production workflows and the NIHAwardExtract template.

### Changed
- **Validation vocabulary is now consistent everywhere.** "Validate & improve" (the action) and "Optimized" (the result) replace the older internal "tune" / "autotune" / "autovalidate" labels across the knowledge-base, extraction, and workflow surfaces and the shared result tiles.
- **Module 6 of the certification is completable again.** The advanced-nodes grader had been checking for task names that never matched what the workflow editor actually stores, so the challenge could never pass. It now reads the editor's real node names and the lab is rebuilt around the Research node (no URL, key, or code required); Code Execution is documented as admin-only. Lesson 3 also drops "thread pool" jargon for plain language about parallel tasks.
- **The version indicator moved out of the always-on header into the account menu**, removing it from the page chrome while keeping it available on demand.
- **Dead/shadow UI removed.** A reachability audit found ~1,758 lines of unreachable frontend (superseded library/extraction panels, redirected pages, orphaned types); they were deleted and a `scripts/dead-code-scan.mjs` added for repeatable checks. No user-facing surface changed.

### Fixed
- **Uploaded documents were never vector-indexed, so KB and project chat retrieved nothing.** Two stacked bugs: per-user Chroma collections were named from the user's email, which contains `@`/`.` (illegal collection characters) and failed every embedding write; and semantic ingestion was dispatched fire-and-forget *racing* the extraction task that writes the document text, so it usually ran against empty text, produced zero chunks, and "succeeded" without retrying — leaving documents permanently unindexed (direct document chat still worked off raw text, which masked it). Collection names are now sanitized (and salted with a hash for uniqueness), and ingestion is chained to run only after extraction has written the text. Diagnosed in the field as 0 of 153 documents having any vector chunks.
- **Project chat could hallucinate document contents, and files moved into a project stayed invisible to it.** Empty KB retrieval was treated like "no KB requested" and fell through to a freeform answer; a dedicated project-KB prompt now refuses to invent project-document contents (while still answering general questions, clearly flagged as such). Moving a document or folder into a project now reindexes it into the project's KB (and removes it from the old one) via a background task.
- **Approval Gate workflows crashed** with "cannot prepare() more than once": the engine walked the dependency graph to execute and walked it again to locate the gate, exceeding graphlib's single-`static_order()` limit. The topological order is now memoized.
- **The admin Workflows search field rejected keystrokes** (its value was reset on every render); it now accepts input while keeping the debounced query.
- **Custom SSO provider labels were ignored.** The public auth-config endpoint read the wrong key, so the login button always fell back to a hardcoded label instead of the admin-configured display name.
- **KB test-query generation timed out as a 502 on larger KBs or slow models.** Generation ran the LLM call inline in the HTTP request, so a slow model or a transient connection blip outlived the proxy gateway timeout and surfaced as a 502. It now dispatches to a Celery worker and the UI polls a new `GET …/test-queries/generate/status` endpoint (mirroring the `add_urls` fix), returning immediately and reading the worker's result by uuid to avoid the incremental-insert race. The worker retries transient `ModelAPIError`/connection blips (10s/20s backoff) while letting genuine 4xx provider errors fail fast, and the status endpoint returns a clean, actionable message instead of leaking a raw exception repr.
- **Background LLM tasks leaked an HTTP client per run, exhausting file descriptors.** Each Celery task runs in a fresh event loop but only closed the loop, not the per-loop `httpx.AsyncClient` cached by `llm_service` — so every LLM-touching task leaked a client and its sockets until the worker hit `[Errno 24] Too many open files` and could no longer reach the model endpoint (surfacing as `ModelAPIError: Connection error.`). Exposed once test-query generation moved onto the worker. A shared `run_task_async` runner now releases the loop's pooled client before tearing the loop down, and the eight LLM-touching task modules route through it.
- **A removed or renamed LLM model gave one user a permanent "Connection error." in KB features.** KB test-query generation, validation judging, and optimization resolved the model from the user's stored selection verbatim; when that model was later deleted or renamed in System Config, the resolver found no endpoint/key and routed to the provider's public default host — which sealed deployments can't reach — a deterministic per-user failure while chat worked fine. These paths now validate the stored pick against the configured models and fall back to the system default (self-healing the stored value), matching chat.
- **KB validation and test-query answers filled empty retrieval from training data.** The RAG prompt on that path never told the model to refuse when retrieval came up empty, so out-of-scope questions got answered instead of a clean "not in the knowledge base." It now carries the same grounding rules as KB chat: treat retrieved chunks as possibly off-topic, refuse rather than guess when context lacks the answer, and prefix any general-knowledge supplementation with "Beyond the retrieved sources:".
- **The automations list issued one query per row to resolve linked workflow/extraction names** (an N+1 that dominated the request). Linked-target names are now batch-resolved in a single `$in` query per type; because the name is decorative, a resolution failure degrades that row to "unnamed" instead of 500-ing the whole `/api/automations` list.
- **Elapsed-time readouts during validation runs** no longer drift or freeze from client/server clock skew or missing UTC offsets, and validation/optimization runs honor cancellation mid-run.
- **Adding URLs to a knowledge base returned a bodyless 502.** `add_urls` awaited the full fetch-and-ingest inline — httpx + Playwright fallback, and up to 50 crawled child pages chunked and embedded serially — which routinely ran past nginx's 300s proxy timeout. The endpoint now sets the source to "building", dispatches the existing async ingestion to a Celery worker, and returns immediately with the queued count (the frontend was already built to poll status); a per-source failure now marks just that source as errored instead of failing the whole request.
- **Source citation pills in chat did nothing on tap and were invisible on touch devices.** The "Sources:" chips under an answer were hover-only `title` tooltips. They're now buttons that toggle an inline panel showing the cited chunk's preview, document title, and page/sheet, with active-state highlighting and `aria-expanded` for accessibility (the desktop hover tooltip is kept).
- **KB test queries and validation history needed a full page reload to appear.** The Validate panel fetched both lists only on mount, so wizard-generated test queries and a just-finished validation run never surfaced live. The Test Queries tab now refetches on entry, and the quality timeline takes a refresh nonce + polling flag so a completed run swaps into History in place without a reload or spinner flash.

### Operator Notes
- **Celery crashes are now observable.** Sentry is initialized in the Celery worker and beat entrypoints (not just the API process) via a shared `init_sentry()` helper, with `CeleryIntegration` and `monitor_beat_tasks` so unhandled task exceptions, soft-time-limit kills, and silently-stopped beat schedules report. Separately, the `engagement_tasks` and `retention_tasks` modules — scheduled in beat but never imported by the worker — are now registered, so data-retention and engagement jobs actually run.
- **Deployments self-report their build version.** `compose.yaml` passes a `VERSION` build-arg to the `api` and `celery` images, and `setup.sh` derives it (explicit env → version file → `git describe` → `dev`) at the `build_image` chokepoint, so `setup.sh`, `--redeploy`, and `--upgrade` all stamp the real version into the footer. An optional `DEPLOYMENT_LABEL` env var distinguishes multiple same-environment deployments. `./run_celery.sh` now resolves the `celery` binary for both `uv`-managed local checkouts and Docker images.
- **`setup.sh --redeploy` / `--upgrade` now prompt for the institution name when it is unset** (persisted to `SystemConfig`, never overwriting an existing value), closing the gap where install skipped it; non-interactive callers are never blocked.
- **This release carries verified-catalog v1.1.1** (creator credit on the ten production workflows and the NIHAwardExtract template) — an additive metadata update with no retirements. Apply it via `./setup.sh --upgrade` / `--seed` or Admin → Catalog.

## [v4.6.0] - 2026-06-10

> **Supersedes v4.5.0.** A `v4.5.0` tag was cut on 2026-06-04, but its release workflow failed during verification, so no images or GitHub release were ever published. Everything intended for v4.5.0 is included below; operators upgrade directly from v4.4.0 → v4.6.0.

### Added
- **Knowledge Base Query workflow node: grounded answers, dynamic queries, visible failures.** The KB Query node gains a mode toggle: **answer** mode retrieves passages and then synthesizes a grounded, citation-bearing answer via LLM (per-claim source labels, explicit "the passages don't contain this" behavior — the same retrieval-honesty rules KB chat uses); **passages** mode returns the raw chunks, now prefaced with a partial-excerpts framing header so downstream LLM steps treat them as evidence rather than complete documents. The query field supports `{{ inputs.output }}` templating (same engine as the API node), so a lookup can be driven by upstream step output — extract a sponsor name, then query that sponsor's policies. Retrieval results carry a 0–1 relevance score (shown as "% match" on citation chips) with an optional minimum-relevance gate, and every failure path — missing KB, empty query, retrieval error, zero hits — surfaces an amber step warning in the run output instead of silently passing empty context to the next step. New steps default to answer mode; existing saved steps keep returning passages.

### Fixed
- **Workflow KB citations never reached the Sources panel.** `MultiTaskNode` (which wraps every task in a real run) dropped the `retrieved_sources` emitted by KB Query steps, so the citations UI only worked in isolated Test Step runs. The wrapper now merges citations — and the new step warnings — through to the run result.
- **Recurring file-descriptor exhaustion from a leaked Redis pub/sub socket** in the browser-automation WebSocket path; the compose `nofile` ulimit was also raised as defense in depth (see Operator Notes).
- **Workflow input-type save failures were silently reverted** in the editor; they now surface an error instead of pretending to save.
- **Blank Formatter template box for pre-rename workflows** (older workflows stored the template under the legacy `prompt` key).
- **KB title not saving on blur.**
- **Zero-token "Connection error" in parallel workflow LLM calls.** Parallel-fanout steps (MultiTaskNode) intermittently failed after ~10s because non-external providers (Insight/Ollama/VLLM) fell back to pydantic-ai's process-wide cached HTTP client, whose connection pool was reused across the per-task event loops — raising a "bound to a different event loop" error the OpenAI SDK re-wrapped as a connection failure before any tokens were sent. Each `get_agent_model()` call now builds a dedicated `httpx.AsyncClient` bound to its own loop. Adds a configurable `workflow_llm_timeout_seconds` (default 120s).

- **Admin panel 403 crash for examiners.** Examiners passed the frontend access gate but every admin analytics endpoint rejects them, and the default Usage tab's `Promise.all` had no `.catch()`, so the 403 surfaced as an uncaught rejection. Examiners are removed from admin-panel access (their workspace is `/verification`), email/teams tabs are hidden from team admins, and the Usage/Users/Teams/Workflows loaders now `.catch()` and show an inline error.

Both now swallow the error and keep prior state; the next refresh recovers. Same hardening applied across other on-mount effects.
- **File-descriptor leak from per-call LLM HTTP and Redis clients.** Building a fresh client per LLM call / per task leaked FDs (toward the `Errno 24` exhaustion seen in production); clients are now reused. (Companion to the earlier per-call Mongo client fix.)
- **Support agents unable to see tickets.** Support-agent status was decided by an exact, case-sensitive match of the configured contact's `user_id` against the account `user_id`, so any mismatch (email vs username, casing, SAML/OAuth UPN) silently set `is_support_agent=false`. Centralized into `support_service.user_matches_support_contact`, matching on `user_id` or email case-insensitively.
- **Demo-user login failing for mixed-case emails.** Demo users were stored with a verbatim mixed-case email, but login lowercases the typed identity before a case-sensitive lookup — so the user got "We couldn't find an account" instead of "wrong password". The demo write paths now normalize email to lowercase; `scripts/normalize_user_emails.py` repairs existing records.
- **Team invites demoting an existing higher-ranked member.** `accept_invite` unconditionally overwrote an existing membership's role, so an owner/admin accepting a lower-role invite was silently demoted. It now only promotes, never demotes — mirroring `accept_join_link`.
- **API Node bodies dropped to zero bytes.** A configured POST/PUT/PATCH body that parsed to a JSON scalar or `null` hit a send path that only handled objects/arrays/strings and was transmitted as zero bytes (a confusing remote 400 with `body=''`). Non-object/array bodies now send their raw JSON text. The API node also defaults its Content-Type to `application/json` for JSON bodies.
- **Share-to-team errors are now diagnosable.** "Failed to clone the underlying workflow or extraction" collapsed two failure paths into one opaque code while the clone helper swallowed exceptions. A new `CloneSourceMissingError` surfaces a distinct `original_missing` (404) with an actionable "refresh the library" message; other clone failures are logged with a real trace.
- **Vanishing "Submit for Verification" dialog.** The modal rendered inline inside the library row's clickable div, so clicks inside it bubbled to the row's `onClick`, opened the editor, and unmounted the modal. It's now portaled to `document.body`.

### Security
- **Admin/credential secret fields no longer attract password-manager autofill.** A fresh user reported their login password populating the OCR API key field (a same-origin `type=password` input). Every admin/credential-vault password field is now hardened with `autoComplete=new-password` plus 1Password / LastPass / Bitwarden ignore hints so the major browsers and managers skip them.
- **Team invite / join-link acceptance is exempt from CSRF.** The accept POST fires immediately after a cross-site SSO redirect, the exact window where the freshly-set CSRF cookie can be briefly unreadable, producing spurious 403s (notably in Outlook/Defender Safe Links webviews). The exemption is safe and mirrors the existing SAML ACS exemption — the unguessable URL token is the actual authorization, so a forged cross-site POST gains nothing an attacker holding the token couldn't already do. Scoped to the accept sub-routes only.
- **Invite acceptance can only promote, never demote** an existing membership (see Fixed), closing a silent privilege-drop.

### Changed

- **KB URL ingest paging Sentry on unreachable hosts.** `ValueError` from `validate_outbound_url` (bad scheme / unresolvable host / private IP) and `httpx.RequestError` (connect/timeout/read errors) are now demoted to warnings in both URL-ingest paths. The source is still marked `status=error` with the message surfaced in the UI; the Celery task stops re-raising since `ValueError` was never in `TRANSIENT_EXCEPTIONS`.
- **Trafilatura library `ERROR` log noise** silenced in `web_fetcher` (empty / unparseable HTML is already handled by the BeautifulSoup fallback). `"Not authenticated"` and `"Rendering cancelled"` added to the frontend Sentry `ignoreErrors`.
- **CSRF errors on multipart uploads.** Several raw `fetch` callsites (support tickets, document uploads, workflow / extraction imports, chat streaming) bypassed `apiFetch` and its 403 self-heal, so users with stale CSRF cookies saw hard errors on those endpoints while JSON requests recovered silently. Centralized through a new `rawFetch` helper; the middleware also logs structured fields on 403 to diagnose remaining reports.

### Security
- **CSRF cookie unspoofable by collision.** The `__Host-csrf_token` prefix is browser-enforced (Chrome refuses to store a cookie of that name without `Secure`, `Path=/`, no `Domain`), so a sibling app on a parent domain or a prior deploy with a different `Path` cannot shadow Vandalizer's token. Header validation against the full multiset of legacy values prevents bypass during the rollout window.
- **Workflow share tokens** grant view-only access only. `can_manage` is forced false for share-token-authenticated recipients, and they see the existing "Make a copy" flow rather than write paths.
- **Author-only message editing**: even support agents can't rewrite someone else's `SupportMessage`.
